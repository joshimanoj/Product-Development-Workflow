# /uat — Human Acceptance Testing (UI Stories Only)

## Purpose

Present the pre-generated UAT checklist (`story-N-uat.md`, produced by `/prd` at planning time) for the human to manually verify the look, feel, and interaction of a UI story on a real device or browser. The human ticks items off in chat. A go-ahead from the human is the gate for merging.

`/uat` does not regenerate the checklist — it loads and presents `story-N-uat.md`. If the file is missing, `/uat` regenerates it from `story-N.md` and logs a warning.

---

## When /uat Runs

| Story Type | After /check | Action |
|------------|-------------|--------|
| UI story (touches any screen or component) | ✅ CI green | Run `/uat` — human must approve before merge |
| Backend only (no UI files touched) | ✅ CI green | Skip `/uat` — merge directly |

**How Claude determines story type:** Reads the Story Type field set in `story-N.md` during `/sprint`. If the field is missing, Claude checks whether any files in the completed task list are UI files (components, screens, views, styles). If yes → UAT required.

---

## What Claude Generates

Claude loads `story-N-uat.md` (pre-generated by `/prd`) and presents it directly. It tells the human exactly what to open, what to do, and what to look for.

If `story-N-uat.md` is missing, Claude regenerates it from `story-N.md` acceptance criteria and user flows, presents it, and logs: "Warning: story-N-uat.md was not found — regenerated now. Check /prd ran correctly for this story."

---

## UAT Checklist Format

```markdown
# UAT Checklist — Story #X: [Story Name]

## Setup

Platform: [Web — open local dev server: http://localhost:3000 | iOS — run on simulator | Android — run on emulator]
Test data: [e.g. use account test@example.com / password: Test1234]
Start at: [e.g. Login screen / Home tab]

## Checklist

### Flow 1: [Happy Path from story-N.md User Flow]

[ ] 1. [Specific action — e.g. Tap 'Sign Up' button]
[ ] 2. [What to verify — e.g. Email field accepts input, keyboard appears]
[ ] 3. [Next action]
[ ] 4. [What to verify — e.g. Success banner appears with correct message]

### Flow 2: [Edge Case / Error State]

[ ] 1. [Action that triggers error — e.g. Submit with empty email]
[ ] 2. [Verify — e.g. Red error message appears below field]
[ ] 3. [Verify — e.g. Form does not submit]

### Visual Check

[ ] Colours match design (primary: #[hex], background: #[hex])
[ ] Typography looks correct (font, size, weight)
[ ] Spacing feels consistent — no elements crowded or misaligned
[ ] Interactive states work — buttons have tap/hover feedback
[ ] Animations feel smooth (transitions, loading states)

### Responsive / Device Check

[ ] [Web: test at mobile width 375px — layout intact]
[ ] [Web: test at desktop width 1280px — layout intact]
[ ] [Mobile: test on small device — no content cut off]

### Edge Cases

[ ] [Story-specific edge case from story-N.md]
[ ] [Story-specific edge case from story-N.md]

## Sign-off

Reply with one of:
✅ APPROVED — all items checked, ready to merge
❌ CHANGES NEEDED — [describe what needs fixing]
```

---

## Human Sign-off in Chat

The human reviews each item on the checklist manually on the real device or browser, then replies in the chat:

| Response | Meaning | Next Step |
|----------|---------|-----------|
| ✅ APPROVED | All items verified, story looks and feels correct | Claude creates PR summary + merge instruction |
| ❌ CHANGES NEEDED | One or more items failed or look wrong | Claude appends the rejection to `uat-log.md`, then loops back to `/dev` for the fix |

---

## After Approval

1. **Confirm PR exists** for this branch. Create one if missing.

2. **Merge to main:**
   ```bash
   gh pr merge [PR-number] --squash --delete-branch
   git checkout main && git pull
   ```

3. **Append a checkpoint to `HANDOVER.md`** — written at merge time, when the story is truly complete:
   ```markdown
   ---

   ## Checkpoint: Story #N [name] | [YYYY-MM-DD] | MERGED ✅

   Story complete: [one sentence — what was built and shipped]

   UAT fixes applied: [list any fixes made between /check and UAT sign-off, or "none"]

   Next: Story #[N+1] [name] | Ready for: /prd
   ```

4. **Update `Sprint.md`:**
   - Mark story `✅` in Story Index
   - Update Active Pointers to point to Story #N+1

5. **Clear `TODO.md`:**
   ```
   # TODO
   _Ready for Story #N+1: [name] — run /prd to generate task list._
   ```

6. **Commit and push to main:**
   ```bash
   git add HANDOVER.md Sprint.md TODO.md
   git commit -m "chore(story-N): UAT approved — mark complete, clear TODO for story #N+1"
   git push origin main
   ```

7. **If this story had UAT rejections**, update those entries in `uat-log.md`: fill in Fix Applied and set Status to Resolved.

8. **Tell the user:** "Story #N complete and merged ✅. Ready to run /prd on Story #[N+1]?"

---

## After Changes Needed

Do not re-run `/uat` immediately. First, append a rejection entry to `uat-log.md` using the format below. Then go back to `/dev` to fix the specific issues. Re-run `/check`. Then re-run `/uat` — it will present the existing `story-N-uat.md` checklist.

```markdown
# uat-log.md — append one block per rejection, never delete entries

---

## UAT Rejection: Story #X — [Story Name] | [YYYY-MM-DD HH:MM]

Attempt: [N]   # increment if same story is rejected multiple times

Failed items:
- [checklist item that failed, copied verbatim from story-N-uat.md]

Human feedback: [exact words the human used]

Root cause (if known): [e.g. wrong token, missing state, responsive rule not applied]

Fix applied: [filled in after /uat passes — one line describing what changed]

Status: Open | Resolved
```
