# /sprint — Transform product_note.md into a Sprint Roadmap

## Purpose

Read the approved `product_note.md`, `architecture.md`, and `Design.md` and produce `Sprint.md` (the roadmap and meta record) plus an individual `story-N.md` file for each user story. `/sprint` does not extract or infer features — it reads them directly from the Features section of `product_note.md`.

---

## Inputs Required

- **`product_note.md`** — status must be Approved
- **`architecture.md`** — status must be Approved
- **`Design.md`** — status must be Approved
- **`/prototypes` folder** — approved HTML prototype(s)
- **`blueprint-[platform].md`** — one per platform declared in `product_note.md`; status must be Approved. Generated by `/design` on prototype approval. If any blueprint file is missing, stop and tell the user: "Blueprint file for [platform] not found. Run /design to generate the full design output suite before running /sprint."
- **`theme.json`**, **`component-behavior-guide.md`**, **`ui-patterns-[platform].md`** — generated by `/design` on prototype approval; required for UI stories.

> **Gate:** If any document is not in Approved status, or if any blueprint file is missing for a declared platform, stop and tell the user. Do not proceed with planning against unapproved inputs.

---

## Emulator Decision

Before planning begins, Claude reads `architecture.md` Section 8 (Testing Strategy) to confirm mobile emulator targets. If the section is incomplete or targets are not confirmed, Claude asks: "This sprint includes UI stories. Will you need iOS Simulator, Android Emulator, or both for testing?" This is confirmed once here and baked into all subsequent `/prd` task specs and CI config. It is not asked again at `/check`.

## Phase 1: Read & Map

### A. Read product_note.md

- Section 4 (Features) → source for all user stories. Each feature becomes one or more stories.
- Section 1 (Vision) + Section 2 (Target Users) → context for writing story descriptions.
- Section 5 (Out of Scope) → rejection list. Do not create stories for anything listed here.
- Section 6 (Constraints) → sprint duration, platform targets, compliance requirements.
- Section 7 (Success Metrics) → Definition of Done criteria at the product level.

### B. Read architecture.md

- Section 1 (Technical Stack) → tech stack for all task specs downstream.
- Section 3 (Data Models) + Section 4 (API Structure) → technical notes for stories.
- Section 5 (Architecture Patterns) → code standards for `/prd` and `/dev`.
- Section 7 (Non-Functional Requirements) → non-functional acceptance criteria for every story.
- Section 8 (Testing Strategy) → test runner, E2E framework, confirmed emulator targets.

### C. Read Design.md

- Section 3 (Information Architecture) → screen and navigation context for UI stories.
- Section 4 (Core User Flows) → user flow steps for story acceptance criteria.
- Section 7 (Interaction Patterns) → UI behaviour expectations.

### D. Read Blueprint Files (UI stories only)

For each platform declared in `product_note.md`, read the full `blueprint-[platform].md` file. Load the complete screen index so that when writing each UI story, the correct screen sections can be identified and embedded. The blueprint is the authoritative screen-level reference — it takes precedence over Design.md where they conflict.

### D. Identify Technical Prerequisites (Sprint 0)

Before any feature sprint, identify setup work that must complete first:

- Infrastructure: dev environment, database, CI/CD pipeline, hosting.
- Third-party: API keys, external service accounts, library evaluation.
- Data foundation: schema creation, migrations, seed data.
- Version control: GitHub repo created, remote configured, CI workflow committed, feature branch strategy confirmed.

```markdown
## Technical Prerequisites (Sprint 0)

Infrastructure: [PostgreSQL, Redis, S3, CI setup]
Third-Party: [Stripe key, SendGrid key, etc.]
Data Foundation: [Initial schema, Prisma migrations, seed data]
GitHub Repo: gh repo create [repo-name] --private --source=. --remote=origin --push
Issue Tracking Files:
  touch bugs.md && echo "# bugs.md — Bug Log\n\nInitialized: $(date +%Y-%m-%d)" > bugs.md
  touch story-change-log.md && echo "# story-change-log.md — Story Change Log\n\nInitialized: $(date +%Y-%m-%d)" > story-change-log.md
Estimated Time: 2-3 days
Blocker: Must complete before Sprint 1
```

## Phase 3: Sprint Planning

### Story Grouping

- Dependencies — what must come first?
- Priority — Must Have features before Should Have.
- Technical Risk — validate uncertain areas early.
- Logical Flow — group stories that share components or data.

### Sprint Structure

```markdown
## Sprint N: [Theme]

Duration: [N weeks]
Goal: [User value delivered]
Success Metric: [How we know it worked]

Stories:
Story #X — [Title] Complexity: [1-5] Type: UI | Backend Priority: High|Med|Low

Capacity: [Total story points]
Risks: [Technical / Resource / Scope risks + mitigations]
DoD: [ ] All stories verified [ ] Demo prepared [ ] Deployed to staging
```

---

## Phase 4: Critical Path

```markdown
## Critical Path

Sprint 1 → Sprint 2 → Sprint 3

Blocking Dependencies:
Story #5 (auth) MUST complete before Story #12 (user profiles)

De-risking:
Build Story #5 first in Sprint 1
Run payment spike before Sprint 2 starts
```

## Output — Sprint.md + story-N.md files + architecture-dev-summary.md

`Sprint.md` is the roadmap and meta record for the life of the project. It is never deleted or cleared — only updated as stories complete. It contains the sprint breakdown, critical path, and a status index of all stories. It does NOT contain individual story detail. Each story is written to its own `story-N.md` file (e.g. `story-1.md`, `story-2.md`). `/prd` loads only the active `story-N.md` file, keeping context lean regardless of sprint count.

`/sprint` also generates `architecture-dev-summary.md` — a focused subset of `architecture.md` containing only what `/dev` needs: stack, naming conventions, file structure, and test runner commands. `/dev` loads this summary instead of the full `architecture.md`.

### Sprint.md Format

```markdown
# Sprint Plan: [Project Name]

Generated: [Date] | Total Stories: [N] | Duration: [X weeks]

Contents:
1. Technical Prerequisites (Sprint 0)
2. Story Index (title, type, status, complexity — links to story-N.md)
3. Sprint Breakdown
4. Critical Path
5. Roadmap

Note: Full story detail lives in story-N.md files (e.g. story-1.md).

Active Sprint: 1 | Next Story: #1 | Ready for: /prd
```

### architecture-dev-summary.md — Prescribed Format

Generated once by `/sprint` from `architecture.md`. Loaded by `/dev` instead of the full `architecture.md`. Contains only what is needed for implementation tasks.

```markdown
# Architecture Dev Summary: [Product Name]

Generated: [Date] | Source: architecture.md v[N]

## 1. Stack

Platform: [from architecture.md Section 1]
Language: [primary language]
Framework: [framework]
State Management: [approach]
Backend: [approach] | Database: [db] | Auth: [auth approach]

## 2. Naming Conventions

Files: [camelCase / snake_case / PascalCase]
Components: [PascalCase]
Variables: [camelCase]
Database columns: [snake_case]

## 3. File Structure

[Exact file structure from architecture.md Section 5]

## 4. Test Runner Commands

Unit: [command from architecture.md Section 8]
E2E: [command]
Lint: [command] | Type check: [command]
```

## Claude Instructions for /sprint

1. Verify `product_note.md`, `architecture.md`, and `Design.md` are all Approved. Stop if not.
2. Confirm emulator targets from `architecture.md` Section 8. Ask if not set.
3. Read Section 4 of `product_note.md` — this is the exact feature list. Do not add or infer features.
4. Read `architecture.md` Sections 1, 3, 4, 5, 7, 8 for technical context.
5. Read `Design.md` Sections 3, 4, 7 for UX and interaction context.
6. Identify Sprint 0 prerequisites.
7. Generate user stories — one per feature entry (or split if complex). Use the template above.
8. Mark Story Type (UI / Backend) for each story — this determines whether `/uat` runs.
9. Group into sprints by dependency, priority, and risk.
10. Identify critical path.
11. Write `Sprint.md` (roadmap/meta: story index, sprint breakdown, critical path, roadmap). Write a separate `story-N.md` file for each user story containing the full story detail from the template above. For every UI story, identify which blueprint screens it touches and paste the verbatim Zones table from `blueprint-[platform].md` for each of those screens directly into the story's Design References section. Do not summarise — copy the table exactly as it appears in the blueprint. This embedded content is what `/prd` uses to generate tasks — it does not re-read the blueprint file separately. Generate `architecture-dev-summary.md` from `architecture.md` — extract only: Section 1 (Platform & Technical Stack), the naming conventions and file structure from Section 5 (Architecture Patterns), and test runner commands from Section 8 (Testing Strategy). This file is used by `/dev` in place of the full `architecture.md`. Present to user for approval.
12. Ask: "Sprint plan ready with [N] stories across [M] sprints. Ready to run /prd on Story #1?"
