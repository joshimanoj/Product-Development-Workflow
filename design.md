# /design — Living UX & Design Document

## Purpose

`Design.md` starts as a draft generated by Claude from `product_note.md` before any prototyping. It is then updated through as many prototype review cycles as needed. Each iteration is tracked with a version entry. When the human approves the prototype, `Design.md` is marked Approved and the full design output suite is generated from it. After that point it is stable — changes require a new version entry.

---

## Iteration Lifecycle

- Claude generates `Design.md` v0.1 from `product_note.md` before prototyping begins.
- Claude builds an HTML prototype based on `Design.md`.
- Human reviews — requests changes or approves.
- If changes requested: Claude updates `Design.md` and rebuilds the prototype. Repeat.
- When human approves: `Design.md` status set to Approved. Full design output suite generated. `/sprint` unlocked.

---

## Prototype Format Rules

The prototype is built for **human review** — to answer "does this feel right to use?" It is not the source for implementation. The design output suite (generated on approval) is what downstream commands build from.

**Web projects:**
- Full browser HTML, responsive
- Desktop and mobile layouts in the same file, toggled by a viewport switcher or separate sections
- Hover states, cursor changes, and keyboard interactions included

**Mobile app projects (Android / iOS):**
- HTML rendered inside a fixed 390×844px phone frame
- App conventions only: no hover states, tap targets ≥ 48dp, bottom navigation, status bar and home indicator rendered
- Touch gestures simulated via click — no mouse-specific behaviour
- Scrollable content within the frame, not the browser page

**Projects with both web and mobile in scope:**
- Two separate prototypes: web first, mobile after web is approved
- Mobile prototype is built after the web prototype is approved — it is not a responsive version of the web prototype, it is a fresh design pass for the mobile form factor

---

## What Design.md Must Contain

**Design.md is not a list of colours and fonts.** It is a document that answers: how does this product feel to use? What mental model does it create? What decisions has the team made about UX that every developer must respect? Thin design docs produce inconsistent, low-quality products.

> **Quality Gate:** Every section is required. Claude must challenge and expand any thin or vague section before marking the document Approved.

## Prescribed Format

```markdown
# Design: [Product Name]

Version: 0.1 | Status: Draft | In Review | Approved | Date: [YYYY-MM-DD]

## Version History

| Version | Date | Changes | Status |
|---------|------|---------|--------|
| 0.1 | YYYY-MM-DD | Initial draft from product_note.md | Draft |
| 0.2 | YYYY-MM-DD | Updated nav pattern after prototype review | In Review |
| 1.0 | YYYY-MM-DD | Approved after prototype sign-off | Approved |

## 1. UX Principles

The 3-5 guiding principles that all design decisions must respect.
These are not generic ("be simple") — they are specific to this product and user.

- [Principle 1: e.g. "Zero learning curve — a first-time user completes their first task in under 60 seconds without reading any instructions."]
- [Principle 2: e.g. "Trust through transparency — users always know what the app is doing with their data and why."]
- [Principle 3: ...]

## 2. User Mental Model

How does the primary user think about this product?
What real-world metaphor does the UI lean on (inbox, dashboard, canvas, feed)?
What do users expect to happen when they open the app?
What is the single most important action in the product?

## 3. Information Architecture

Top-level navigation structure:
[Tab/Screen 1] — purpose and what lives here
[Tab/Screen 2] — purpose and what lives here
[Tab/Screen 3] — purpose and what lives here

Navigation pattern: [Tab bar / Drawer / Stack / Bottom sheet / etc.]
Rationale: [Why this pattern for this user and product]
Modal vs push navigation: [Rules for when to use each]

## 4. Core User Flows

### Flow 1: [Name — e.g. Onboarding]

Trigger: [What starts this flow]
Steps:
1. [Screen/state] → [User action] → [Result]
2. [Screen/state] → [User action] → [Result]
Success state: [What the user sees/feels when done]
Failure / error state: [How errors are handled]

### Flow 2: [Name — e.g. Core task completion]

[Repeat structure]

## 5. Visual Direction

Tone: [Professional / Playful / Minimal / Bold / Calm / Energetic]
Personality: [3 adjectives that describe how this product should feel]
Inspiration: [Apps or products whose aesthetic we are drawing from and why]
Layout density: [Spacious / Balanced / Dense — and why for this user]
Typography approach: [Functional / Expressive — headline vs body hierarchy]
Imagery / illustration: [Photography / Icons only / Illustrations / None]

Note: Visual Direction is the intent that drives the design. The precise rules derived
from it — colour usage, typography hierarchy, iconography style — are documented in
`style-guide.md`, generated on approval.

## 6. Accessibility

Minimum contrast ratio: [e.g. 4.5:1 for body text]
Touch target minimum size: [e.g. 44x44pt iOS, 48x48dp Android]
Screen reader support: [required / not required — if required, approach]
Dynamic type (iOS) / font scaling (Android): [supported / not supported]

## 7. Responsive / Adaptive Rules

[Web: breakpoints and layout changes at each]
[Mobile: how layout adapts to small vs large phones]
[Tablet: supported / not supported — if yes, approach]

## 8. Blueprint Change Management

**Post-approval changes:** After blueprint status is Approved, it is the authoritative record of UI decisions. Any change to a screen — however small — requires: a new version entry in the blueprint, human approval of the changed section, and the relevant story's Design References section updated before `/prd` runs. No UI change is valid until it is in the blueprint.

## 9. Open Design Decisions

Unresolved questions that may affect implementation.
(Clear this section when Status = Approved)
- [Decision needed | Options | Owner | Resolve by]
```

## Design Output Suite — generated on approval

Once `Design.md` is marked Approved, Claude reads `product_note.md` to determine the declared platforms and generates the relevant outputs from the approved prototype. Outputs are generated by Claude reading the prototype source carefully — every zone, every component, every string, every value — not by summarising intent.

These files are the authoritative reference that all downstream commands (`/sprint`, `/prd`, `/dev`) build from. The prototype HTML is for human review only — it is not read by downstream commands.

### Platform → Output Matrix

| Output | Web | Android | iOS |
|---|---|---|---|
| `style-guide.md` | ✓ | ✓ | ✓ |
| `theme.json` | ✓ | ✓ | ✓ |
| `asset-catalogue.md` | ✓ | ✓ | ✓ |
| `asset-library.html` | ✓ (if custom assets) | ✓ (if custom assets) | ✓ (if custom assets) |
| `component-behavior-guide.md` | ✓ | ✓ | ✓ |
| `blueprint-web-desktop.md` | ✓ | — | — |
| `blueprint-web-mobile.md` | ✓ | — | — |
| `blueprint-android.md` | — | ✓ | — |
| `blueprint-ios.md` | — | — | ✓ |
| `ui-patterns-react.md` | ✓ | — | — |
| `ui-patterns-react-native.md` | — | ✓ (if RN) | — |
| `ui-patterns-compose.md` | — | ✓ (if Kotlin) | — |
| `ui-patterns-swiftui.md` | — | — | ✓ |

For Android: Claude asks "React Native or Kotlin?" before generating the UI patterns file. One is generated, not both.

`blueprint-android.md` and `blueprint-ios.md` share the same UX content (screens, flows, copy) but are authored separately — Android uses Android platform conventions, iOS uses iOS platform conventions. They are not copies of each other.

When a project adds a new platform later (e.g. web is done, Android pass begins), only the outputs for the new platform are generated. Existing outputs are not regenerated unless `Design.md` has been revised.

## Output File Specifications

### `style-guide.md`

The visual language rulebook. Tells every developer and Claude instance *when* and *how* to use the design system — not just what the values are.

```markdown
# Style Guide: [Product Name]

## Typography Hierarchy
For each text role: which font family, size token, weight token, line height token, when to use it.
e.g. Screen Title: display / 2xl / extrabold / tight — used for top-level screen headings only

## Colour Usage Rules
For each colour token: what it is for, what it is never used for.
e.g. color.brand.primary — CTAs, active nav indicators, links. Never used for body text or backgrounds.

## Iconography
Icon library: [e.g. Lucide, Material Symbols, SF Symbols]
Style: [Outlined / Filled / Rounded — one style only, no mixing]
Sizes: [small: 16dp / medium: 24dp / large: 32dp — when to use each]
Colour: [always inherits text colour / always brand.primary / contextual rules]

## Imagery & Illustration
Placeholder treatment: [gradient + emoji / grey rectangle / blurred thumbnail]
Aspect ratios: [thumbnails: 16:9 / avatars: 1:1 / hero: 4:3]
Image style: [photography / illustration / emoji-led — rationale]

## Spacing Philosophy
Base unit: [e.g. 8dp]
Card internal padding: [rule]
Screen edge margins: [rule]
Vertical rhythm: [rule]

## Do / Don't
[3-5 specific rules with a correct and incorrect example each]
e.g. DO: use radius.2xl on primary CTAs. DON'T: mix radius sizes within the same card.
```

### `theme.json`

All design tokens as values. No usage rules — those live in `style-guide.md`. This file is the single source of truth for all token values across all platforms.

```json
{
  "color": {
    "brand": { "primary": "#hex", "secondary": "#hex", "accent": "#hex" },
    "surface": { "background": "#hex", "card": "#hex", "overlay": "rgba(...)" },
    "text": { "primary": "#hex", "secondary": "#hex", "inverse": "#hex", "muted": "rgba(...)" },
    "semantic": { "success": "#hex", "successSurface": "#hex", "error": "#hex", "warning": "#hex" },
    "gradient": { "[name]": "[css gradient value]" }
  },
  "typography": {
    "family": { "display": "[font name]", "body": "[font name]" },
    "size": { "xs": "11px", "sm": "12px", "base": "14px", "md": "15px", "lg": "17px", "xl": "20px", "2xl": "26px", "3xl": "36px", "4xl": "42px" },
    "weight": { "regular": 400, "semibold": 600, "bold": 700, "extrabold": 800 },
    "lineHeight": { "tight": 1.1, "normal": 1.2, "relaxed": 1.6 }
  },
  "spacing": { "1": "4px", "2": "8px", "3": "12px", "4": "16px", "5": "20px", "6": "24px", "8": "32px", "10": "40px", "12": "48px" },
  "radius": { "sm": "8px", "md": "14px", "lg": "16px", "xl": "18px", "2xl": "24px", "full": "9999px" },
  "shadow": { "[name]": "[css shadow value]" },
  "elevation": { "none": 0, "low": 2, "mid": 4, "high": 8 }
}
```

Units in `theme.json` are always `px` for web. Platform-specific unit conversion (dp for Android, pt for iOS) is handled by `ui-patterns-[platform].md` — not by maintaining separate token files.

### `asset-catalogue.md`

Every visual asset in the product, named and described. The single source of truth for icons, images, gradients, and illustrations. Downstream commands never invent or guess assets — they reference this catalogue by name.

The `Source` column tells Claude exactly where to get the actual asset:
- `asset-library.html#[id]` — custom asset, extract base64 data from the library file by matching `id`
- `Lucide / Material Symbols / SF Symbols` — standard icon library, use by name in code, no file needed
- `theme.json → [token path]` — code-defined visual (gradient, shadow, colour), generate from tokens, no file needed

```markdown
# Asset Catalogue: [Product Name]

## Icons
| Name | Library | Style | Sizes used | Colour rule | Screens used | Source |
|---|---|---|---|---|---|---|
| chevron-left | Lucide | Outline | 24dp | text.primary | All back buttons | Lucide |
| play-circle | Lucide | Filled | 32dp, 48dp | text.inverse | Video card, player | Lucide |

## Images & Placeholders
| Name | Type | Dimensions | Treatment | Screens used | Source |
|---|---|---|---|---|---|
| creator-photo | JPEG | 130×130px | Circle crop, 3px border rgba(255,255,255,0.5) | Splash | asset-library.html#creator-photo |
| video-thumbnail | Remote URL | 160×90dp (16:9) | Rounded radius.lg, gradient fallback | Video Library | asset-library.html#video-thumbnail-placeholder |

## Gradients
| Name | Value | Screens used | Source |
|---|---|---|---|
| splash-bg | linear-gradient(135deg, #F43F5E, #9333EA, #EC4899) | Splash background | theme.json → color.gradient.splash-bg |
| hero-purple | linear-gradient(160deg, #EDE9FE, #FDF4FF) | OB-1 hero zone | theme.json → color.gradient.hero-purple |

## Illustrations & Emoji

## Dark Mode Deltas
(Only populated if dark mode is in scope per product_note.md)

|---|---|---|
| surface.background | #FFFFFF | #0F0F0F |
| text.primary | #1E1B4B | #F9FAFB |
```

### `asset-library.html`

Contains every custom asset (images, illustrations, custom icons) as named, base64-encoded entries. Generated only when the product has custom assets that cannot be sourced from a standard icon library or generated from tokens.

The asset name is the shared key between `asset-catalogue.md` and `asset-library.html`. When `/dev` needs a custom asset, it looks up the name in `asset-catalogue.md` to get dimensions and treatment rules, then extracts the base64 data from `asset-library.html` by matching the `id` attribute.

```html
<!DOCTYPE html>
<html>
<head>
  <title>Asset Library: [Product Name]</title>
  <style>
    body { font-family: sans-serif; padding: 24px; }
    .asset { margin-bottom: 32px; border-bottom: 1px solid #eee; padding-bottom: 24px; }
    .asset-name { font-weight: bold; font-size: 14px; margin-bottom: 8px; }
    .asset-meta { font-size: 12px; color: #666; margin-bottom: 8px; }
    img { display: block; }
  </style>
</head>
<body>

  <div class="asset">
    <div class="asset-name">creator-photo</div>
    <div class="asset-meta">Type: JPEG | Dimensions: 130×130px | Used: Splash screen</div>
    <img id="creator-photo" src="data:image/jpeg;base64,[base64 data]" width="130" height="130" />
  </div>

  <div class="asset">
    <div class="asset-name">video-thumbnail-placeholder</div>
    <div class="asset-meta">Type: PNG | Dimensions: 160×90px | Used: Video Library fallback</div>
    <img id="video-thumbnail-placeholder" src="data:image/png;base64,[base64 data]" width="160" height="90" />
  </div>

</body>
</html>
```

**Asset library rules:**
- Every `id` attribute must exactly match the name in `asset-catalogue.md` Source column
- Only custom assets live here — standard icons and token-defined visuals are never embedded
- One entry per asset — no duplicates
- Each entry includes a `asset-meta` div with type, dimensions, and screens used — matching `asset-catalogue.md`
- If a custom asset is not yet available at design time, create a placeholder entry with a generated gradient or solid colour block and note it as `[PLACEHOLDER — replace before dev]`

### `blueprint-[platform].md`

Screen-by-screen specification of every screen in the product for a specific platform. Written by Claude reading the approved prototype source line by line — every zone, every component, verbatim copy, exact measurements in platform-native units, token references (never raw values).

One file per platform: `blueprint-web-desktop.md`, `blueprint-web-mobile.md`, `blueprint-android.md`, `blueprint-ios.md`.

```markdown
# Blueprint: [Platform] — [Product Name]
Version: 1.0 | Status: Approved | Date: YYYY-MM-DD
Source: Extracted from approved prototype by Claude reading source directly.
Platform units: px (web) | dp (Android) | pt (iOS)

1 · Sign In
2 · Home
[etc.]

| State | Trigger | What changes |
|---|---|---|
| Loading | Screen mount | Skeleton on content zones, content hidden |

| Error | API failure | Error toast visible, retry button visible |

### Navigation
| Action | Result |
|---|---|
| Tap [component] | Push → Screen [N] [Name] |
| Tap back / Back button | Pop → Screen [N] [Name] |
| [Platform gesture] | [Result] |

### Platform Notes
```

**Blueprint rules — strictly enforced:**
- All text strings are verbatim from the prototype — never paraphrased, never invented
- All measurements are in platform-native units — never CSS pixels in a mobile blueprint
- All colour and style references use token names from `theme.json` — never raw hex values
- Every screen documents loading, empty, and error states — not just the happy path
- Assets reference `asset-catalogue.md` by name — never described inline
- `blueprint-android.md` and `blueprint-ios.md` are authored separately with platform-native conventions — they are not copies of each other

## Claude Instructions for /design

1. Read `product_note.md` fully — users, features, constraints, tone — before drafting anything.
2. Generate `Design.md` v0.1 as a draft. Do not ask the human to fill it in — Claude drafts the full document and presents it for review.
3. Build an HTML prototype based on `Design.md`. Follow Prototype Format Rules above — phone-framed for mobile projects, full browser for web projects.
4. After human review, update `Design.md` with a new version entry and rebuild the prototype. Repeat until approved.
5. Challenge thin sections: "You've described the visual direction as 'clean' — can you give me 3 specific apps whose aesthetic you're drawing from and why?"
6. Only mark Status: Approved when the human explicitly approves the prototype.
7. On approval: read `product_note.md` to determine declared platforms. For Android, ask "React Native or Kotlin?" before proceeding. Then generate all outputs in the platform matrix above. Read the prototype source directly and carefully — every zone, string, measurement, and asset. Do not summarise or paraphrase — extract exactly.
8. Generate outputs in this order: `theme.json` first (all other files reference it) → `style-guide.md` → `asset-catalogue.md` → `asset-library.html` (if custom assets exist) → `component-behavior-guide.md` → `blueprint-[platform].md` files → `ui-patterns-[platform].md` files.
9. Every blueprint string must be verbatim from the prototype. Every spec value must use a token name from `theme.json`, not a raw value. Every asset must be named and listed in `asset-catalogue.md`. If any string, value, or asset is ambiguous or missing from the prototype, flag it as a gap and confirm with the human before marking outputs complete.
10. When a new platform pass begins (e.g. Android pass after web is complete): generate only the outputs for that platform. Do not regenerate existing outputs unless `Design.md` has been revised since they were last generated.
11. `/sprint`, `/prd`, and `/dev` read the blueprint and ui-patterns files for the active platform — not the prototype HTML. `/sprint` embeds the relevant blueprint screen sections directly into each story's Design References section at story generation time. The prototype is never referenced by downstream commands.
